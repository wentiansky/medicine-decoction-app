# 项目协作约定

- 用户每次提出改动或新需求时，必须先用自己的话复述需求、澄清关键细节，并等用户确认理解无误后，才能开始修改代码；只有用户明确表示“直接改”“不用确认”或需求只是纯文档记录等低风险操作时，才可以跳过确认。
- 修改文件前先检查工作区状态；如果发现用户已有未提交改动，必须保留并围绕现有内容增量修改，不能直接覆盖或回滚。
- 后续生成 commit message 时，类型前缀可以使用英文规范，例如 `feat`、`fix`、`docs`；冒号后的修改内容描述必须使用中文。
- 代码修改完成后，默认不要构建或打包安装包。
- 当用户输入 `BA` 时，表示需要执行 Android 打包；此时可以运行 `./gradlew :app:assembleDebug :app:assembleRelease` 这类 Android 打包命令，并按项目约定把 APK 复制到 `dist/` 目录。
- 当用户输入 `BI` 时，表示需要执行 iOS 打包；执行前先检查当前项目是否已有 iOS 打包配置和签名配置，再按项目约定打包。
- 日常开发完成后，优先使用轻量级校验，例如 `npm test` 和 JS 语法检查。
- 当用户输入 `GP` 时，表示需要提交代码并推送到已配置的 git 远程仓库。
