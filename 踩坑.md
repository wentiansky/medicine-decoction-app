# 熬中药计时器踩坑记录

## 1. 只弹出应用内到期弹窗，未弹出系统闹钟提醒

### 遇到的问题

计时器倒计时结束后，只有 App 内部的 `Alert` 弹窗提示。只要用户不盯着 App，或者 App 进入后台，就很容易错过熬药阶段完成提醒。

### 原因

最初的提醒逻辑主要依赖 JS 层计时和应用内弹窗。应用内弹窗只能在 App 前台可见时发挥作用，不能作为系统级提醒，也无法保证在后台准时唤醒用户。

另外，Expo/React Native 的普通通知能力可以作为提醒兜底，但在 Android 真机上，普通通知不等于“闹钟级提醒”。如果需要更接近系统闹钟的准时触发，需要进入 Android 原生层使用 `AlarmManager`。

### 解决方案

新增 Android 原生闹钟模块：

- 使用 `AlarmManager.setAlarmClock` 调度阶段到期提醒。
- 到点后通过原生广播接收器处理提醒。
- 创建高优先级通知渠道，保留通知栏提醒。
- 增加原生日志，方便从 App 的日志页看到调度、广播、通知、闹钟页面/浮窗是否执行。

同时保留 JS 层的应用内提示作为前台兜底，但不再把它当作后台提醒的主路径。

### 取舍和考量

- `AlarmManager` 比 JS 定时器可靠，适合熬药这种不能错过的场景。
- 当前原生工程使用 `USE_EXACT_ALARM` 和 `AlarmManager.setAlarmClock` 作为闹钟调度主路径，不再把“精确闹钟权限”作为用户需要在 App 内逐步开启的权限项。
- 普通通知继续保留，因为它是浮窗失败或用户错过浮窗时的重要兜底。

## 2. App 内能弹系统闹钟，切后台后不能弹出系统闹钟提醒

### 遇到的问题

在 App 内部测试时，闹钟提醒能够出现。但切到后台，或者用户正在使用其他 App 时，到点后只看到通知栏通知，没有全局弹出闹钟提醒。

日志显示：

- 原生闹钟已经成功调度。
- 广播已经准时收到。
- 通知已经成功发送。
- 从广播里请求启动闹钟 Activity 也执行了。
- 但真正的提醒页面没有覆盖到当前正在使用的 App 上方。

### 原因

Android 对“后台启动 Activity”限制越来越严格。即使 App 具备闹钟调度能力、通知权限和全屏提醒能力，不同 Android 版本和不同厂商 ROM 仍可能拦截后台 Activity 直接覆盖当前界面。

也就是说，问题不是闹钟没触发，而是“从后台把一个普通 Activity 拉到用户当前正在使用的 App 上方”不可靠。

### 解决方案

把“系统闹钟页面”方案调整为“浮窗闹钟”方案：

- 本应用内显示闹钟提醒页，不需要额外权限。
- 切到后台或在其他应用之上显示闹钟提醒页，需要打开“显示悬浮窗”权限。
- 有悬浮窗权限时，实际提醒路径是“悬浮窗 + 闹钟通知”并存；用户即使关掉悬浮窗，仍可通过通知再次进入闹钟提醒页。
- 无悬浮窗权限时，后台提醒以高优先级闹钟通知为主，通知尽量以屏幕顶部横幅形式展示(fullScreenIntent)，点击后进入 `AlarmAlertActivity`。
- 锁屏显示闹钟提醒页时，会继续使用锁屏 Activity，并尽量附带全屏通知能力；部分机型仍可能需要额外的“后台弹出界面”或“锁屏显示”权限。
- 新增 `SYSTEM_ALERT_WINDOW` 权限，也就是“显示在其他应用上层”。
- 到点后由原生广播启动 `AlarmOverlayService`。
- `AlarmOverlayService` 使用浮窗覆盖当前界面，显示闹钟提醒。
- 浮窗服务播放闹钟声、持续震动，并提供“我知道了”和“打开应用”按钮。
- 无论是否有悬浮窗权限，正式闹钟通知都会保留，作为兜底和可回溯入口。
- 正式闹钟通知改为一次性通知，点击后自动消失，不再长期常驻通知栏。

### 取舍和考量

- “显示在其他应用上层”权限对用户来说比较敏感，但它正是实现“正在使用其他 App 时也弹出提醒”的关键权限。
- Android 的 `heads-up`/横幅展示并不完全由代码决定，还会受到通知渠道、系统设置和厂商 ROM 的影响，所以无悬浮窗场景下只能“尽量提高弹出概率”，不能保证所有机型都完全一致。
- 浮窗方案比后台拉 Activity 更符合当前 Android 的限制模型，也更可控；无悬浮窗场景则收敛为“高优先级通知 + 用户点击进入闹钟页”的主流路径。
- 通知栏通知既是入口也是兜底，不再只是“回到 App”的普通跳转。

## 3. 权限交互优化

### 遇到的问题

早期版本把缺失权限直接铺在首页卡片里，用户需要手动理解每个权限的含义，再逐个去系统设置里打开。这个体验偏调试工具，不像面向普通用户的主流 App。

特别是 Android 的一些关键权限无法通过一个普通系统弹窗直接授权，例如：

- 显示在其他应用上层权限。
- 部分厂商系统里的后台弹出界面、锁屏显示权限。

用户必须跳转到系统设置页手动开启。如果 App 只是简单列出权限，用户很容易觉得麻烦，甚至直接放弃使用。

### 原因

Android 权限分两类：

- 通知权限这类可以通过系统授权弹窗完成。
- 显示在其他应用上层这类高风险权限，必须跳到系统设置页，由用户手动开启。
- 后台弹出界面、锁屏显示这类厂商权限没有统一 Android API，不同系统命名和入口都可能不同，也不适合让用户在 App 内手动勾选“我已开启”。

因此，不能完全做成“一键同意后全部正常”，只能把流程包装得更清楚、更短、更像引导。

### 解决方案

新增权限引导交互：

- 缺权限时弹出“开启可靠提醒”引导弹窗。
- 弹窗按场景拆解：步骤 1 是后台/其他应用提醒，需要悬浮窗；步骤 2 是锁屏提醒，需要按需补充后台弹出界面和锁屏显示。
- 如果系统检测到悬浮窗已开启，就不再展示步骤 1；锁屏提醒仍保留，但标题去掉“步骤 2”，避免用户看到孤立的第二步。
- 首页保留轻量常驻入口：权限小图标、一句短文案、一个“开启”按钮，避免用户只开启一个权限后就找不到入口。
- 不再要求用户手动勾选“我已开启后台弹出界面/锁屏显示”，因为这类确认可以被绕过，反而会制造假状态。
- 用户从系统设置页返回 App 后，自动重新检测悬浮窗权限；如果悬浮窗已开启，权限弹窗可以关闭，后续锁屏相关权限通过首页入口按需补充。
- 开始/重新开始计时时，即使权限不足也不阻塞流程，只记录日志并继续计时。

### 取舍和考量

- 无法绕过 Android 对特殊权限的系统设置限制，只能优化解释和步骤。
- 不再强制要求全屏提醒权限，减少用户需要开启的权限数量。
- 权限引导的核心从“罗列权限”改成“解释场景”：后台/其他 App 看悬浮窗，锁屏/息屏看厂商锁屏权限。
- 锁屏相关权限不好检测，所以产品上只保留入口和说明，不做虚假的完成状态。
- 首页保持轻量，权限问题只用一行入口承载，降低用户心理负担。

## 4. Android 返回手势不能直接退出子页面

### 遇到的问题

Android 手势左滑或返回键如果不拦截，可能在主页直接退出应用，也可能在子页面或弹窗场景下表现不符合用户预期。

本项目虽然没有引入导航栈，但有日志页、设置弹窗、权限弹窗这些页面状态。用户期望是：

- 有弹窗时，返回先关闭弹窗。
- 在日志页时，返回回到主页。
- 在主页时，第一次返回只提示，再次返回才退出。

### 原因

React Native 默认的 Android 返回行为更接近系统级退出。如果应用内部用状态切换子页面，而不是导航库的 Stack，就需要自己定义返回优先级。

### 解决方案

新增 `getAndroidBackAction` 纯函数，把返回行为收敛成明确顺序：

1. 关闭设置弹窗。
2. 关闭权限弹窗。
3. 从日志页返回主页。
4. 主页第一次返回 Toast 提示“再按一次退出应用”。
5. 2 秒内第二次返回才调用 `BackHandler.exitApp()`。

### 取舍和考量

- 纯函数便于测试，不把所有判断散落在 `App.js` 里。
- 这套行为接近主流 Android 应用，能降低误退出概率。
- 后续如果新增子页面，也应该先接入这个返回优先级，而不是让系统直接退出。

## 5. 真机调试日志太多，需要分页和复制

### 遇到的问题

Android 原生闹钟链路包含调度、广播、通知、Activity、Overlay Service、响铃、振动、权限检测等多个节点。真机调试时日志很快变长，全部渲染会影响查看效率，也不方便只复制关键片段。

### 原因

提醒链路跨 JS 和 Android 原生层，单靠控制台输出很难复盘。尤其是用户反馈问题时，需要看到 App 内记录的原生日志和 JS 日志。

### 解决方案

- 原生侧通过 `AndroidAlarmDebugLog` 写入关键事件。
- App 回到前台时同步原生日志到应用日志。
- 日志页增加分页展示、加载更多、多选复制。
- 清空或重新加载日志时重置分页和选择状态。

### 取舍和考量

- 日志页面不是用户主功能，但是真机适配阶段非常关键。
- 分页能避免日志多时页面变重。
- 多选复制能减少反馈成本，方便只贴相关时间段。
